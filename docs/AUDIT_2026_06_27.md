# KRY Audit — 2026-06-27

## Scope and method

This review is a static architecture, security, claims, and release-readiness audit of the public repository on `main`, with low-risk documentation fixes staged on the `audit-2026-06-27` branch.

Reviewed surfaces:

- package metadata and stdlib build backend;
- GitHub Actions, CodeQL, release verifier, and attribution guard;
- token ledger, mint chain, attestation builder, standalone verifier, provider reconciliation, settlement, and artifact tooling;
- privacy gates, claims boundary, readiness rubric, security policy, and legal posture.

This pass did not run the local test suite in a checked-out working tree. Treat `python3 scripts/kry_release_verify.py --full` as the required validation command before merging any behavioral hardening.

## Executive verdict

No P0/P1 issue was found in the reviewed core. The repository is unusually mature for a research-grade proof/accounting artifact: it has a clear separation of integrity, magnitude, and veracity; a standalone verifier; strict JSON handling; dated magnitude arithmetic; settlement rollback/double-spend guards; public claims boundaries; and a release gate.

The highest-impact next code change is stricter receipt identity validation across all verification surfaces. `hash_version >= 6` already binds `receipt_id` into the public block, but the verifiers should also reject duplicate, missing, or non-monotonic receipt identities explicitly so promotion overlays cannot become semantically ambiguous even when hashes still validate.

The second highest-impact change is wording hardening in module docstrings. The public-facing README and claims docs are careful, but a few internal docstrings still use stronger phrases such as “trustless” or “event occurred” where the correct scope is “internally valid, conserved, magnitude-checkable, and labeled by veracity tier.”

## Round-by-round review

### Round 1 — Repository shape and architecture

The architecture has a coherent flow: usage/routing events become savings report entries; optional minting writes receipts into the hash chain; an attestation exposes content-free public links; a standalone verifier recomputes integrity, conservation, magnitude, and veracity surface; settlement consumes verified balance subject to registry/lease controls.

Assessment: sound separation of concerns. The code is easier to audit because core modules are small and scripts are explicit operator/auditor tools.

Recommended action: keep the current split between `src/kry/` core and `scripts/` verifier/operator tooling. Do not collapse the standalone verifier into package imports.

### Round 2 — Packaging and dependency surface

The package declares zero runtime dependencies and a custom stdlib-only build backend. Optional crypto extras are isolated from the default package. Dev tooling is pinned.

Assessment: strong supply-chain posture. The custom backend is compact and auditable. The tradeoff is that the backend must continue to track wheel metadata requirements manually.

Recommended action: add a release-gate assertion that builds a non-editable wheel and installs it in a fresh venv, then verifies import and script-level smoke commands.

### Round 3 — CI and release gates

CI runs lint, tests, reproducibility smoke, packet workflow smoke, release verifier, and scheduled full reproducibility. Actions are pinned by SHA and default permissions are read-only. CodeQL and attribution guard are present.

Assessment: strong baseline. The only optional improvement is switching critical egress controls from audit mode to block mode after the allowlist is stable.

Recommended action: keep daily/scheduled runs. Add a branch-protection requirement that the packet gate and full release verifier artifacts remain available for every release tag.

### Round 4 — Token ledger accounting

The token ledger validates finite numbers, rejects non-finite JSON constants, saves every earn/spend, and uses cross-process locking plus delta merge. Negative balance is clamped under lock to preserve the `earned - spent == balance` invariant.

Assessment: much improved from a naive ledger. The silent `load_or_create()` reset on read failure is acceptable for a recoverable local cache only because the mint chain can be the source of truth, but it should log a warning.

Recommended action: warn on corrupt ledger fallback and explicitly direct operators to `reconcile_ledger_from_chain()`.

### Round 5 — Mint chain and hash format

The mint chain validates finite inputs, applies supply decay for repeated evidence, uses fsync on append, stores a local tip checkpoint, and has v4/v5/v6 public-block hashing. v6 binds `receipt_id`; v5 gives language-neutral float encoding; v4 binds the public economic block.

Assessment: the core hash model is strong. The remaining gap is semantic validation, not cryptographic hashing: verifiers should enforce strict receipt identity rules rather than only binding whatever identity appears.

Recommended action: for `hash_version >= 6`, reject duplicate receipt IDs, missing receipt IDs, and receipt IDs that do not match their physical sequence (`KRY-00000001`, etc.).

### Round 6 — Attestation and standalone verifier

The attestation is content-sealed and exposes only hashes, event types, timestamps, public economic inputs, tiers, and receipt IDs. The standalone verifier imports no package code and carries its own reference constants.

Assessment: excellent verifier independence. The duplicated verifier logic is intentional and valuable, but it means any new invariant must be added to both package and standalone verifiers with drift tests.

Recommended action: add identical strict-sequence/receipt-id checks to `src/kry/kry_attest.py`, `src/kry/kry_mint.py`, and `scripts/kry_verify.py`, then add a test that mutates an attestation and confirms both verifiers reject it.

### Round 7 — Magnitude and price arithmetic

Magnitude recomputation checks that `kry_minted = tokens_saved × earn_rate × published_multiplier`, including pairwise net multipliers for paid displacement. The unknown avoided-model multiplier is a conservative floor rather than full frontier value.

Assessment: sound. The project correctly separates dated magnitude from current pricing. External users will still need a stronger price source manifest if “current savings” language is used in sales/compliance settings.

Recommended action: move price provenance into a versioned JSON manifest with source references, retrieval date, and manifest hash; bind the manifest hash into release artifacts.

### Round 8 — Veracity tiers and anchors

The code and docs clearly distinguish self-reported, holdout-validated, provider-metered, TLS-notarized, and TEE-attested tiers. The chain proves integrity, not event truth. Chain-head anchors and registry anchors are provided for external publication.

Assessment: honest and unusually explicit. The main remaining risk is operator workflow: an anchor handed over at verification time proves nothing.

Recommended action: external artifacts should require an out-of-band published anchor reference, publication date, medium, and expected hash before any external reliance claim is allowed.

### Round 9 — Provider reconciliation

T1 provider reconciliation supports per-request matching and aggregate billing checks with bounded tolerance. It correctly treats empty reconciliation as undefined rather than perfect agreement.

Assessment: good. Per-request reconciliation is much stronger than aggregate reconciliation; the artifact should surface this distinction prominently.

Recommended action: expose reconciliation mode in the claim register and require stricter language for aggregate-mode packets.

### Round 10 — Savings report and holdout measurement

The savings report uses a minimum holdout sample before assigning `holdout_validated`, applies Wilson lower bound to measured classes, and leaves thin/no-holdout classes self-reported. Displacements are netted against the served model.

Assessment: strong self-restraint. The biggest misuse risk is interpreting full-value self-reported cache hits as externally verified savings.

Recommended action: in external-facing templates, require a prominent line item that separates `self_reported`, `holdout_validated`, and `provider_metered` dollar amounts.

### Round 11 — Settlement and double-spend controls

Settlement validates offers, verifies attestations, applies an in-process reservation, rechecks the registry at commit time under a cross-process lock, supports a default-off shared lease authority for cross-node racing, and labels conservation as measured vs self-asserted.

Assessment: strong for single-host/federated use; not fully trustless global settlement. Internal docstrings should not call it “trustless” without immediately scoping that claim.

Recommended action: change wording to “federated verifier-mediated settlement” and make cross-node lease/anchor requirements explicit in any external settlement examples.

### Round 12 — Artifact privacy and packet gates

The artifact tooling hashes inputs, binds manifests, checks public packet privacy, scans usage/provider exports for private fields, rejects unrecognized string/dict/list fields in bounded logs, and enforces distinct outside-review/buyer/legal channels.

Assessment: strong packet discipline. The privacy allowlist should be maintained conservatively; new provider export fields should start blocked until reviewed.

Recommended action: add regression fixtures for common private-field synonyms such as `text`, `query`, `payload`, `raw`, and provider-specific envelope fields.

### Round 13 — Readiness, claims, and legal posture

The claims boundary clearly blocks external verified-savings, production-ready, tradeable-token, certified-carbon-credit, and buyer-reliance claims until required evidence exists. Readiness is computed from an explicit ladder and keeps A+/production readiness external.

Assessment: strong. A small 13-vs-14 implemented-count drift was corrected in this branch.

Recommended action: make `docs/CLAIMS_BOUNDARY.md` the source of truth for all README/release wording, with a test that forbidden phrases do not appear outside approved contexts.

### Round 14 — Optional crypto, TEE, TLSNotary, and PQC

Optional crypto tiers are correctly treated as authenticity or stronger evidence add-ons, not as default stdlib requirements. Public-key pinning and out-of-band trust remain necessary.

Assessment: correct scope. Do not let optional signatures imply veracity. A self-signed artifact proves only possession of the embedded key unless the verifier supplies a pinned key.

Recommended action: ensure every optional-crypto success output says exactly what it proves and what it does not prove.

### Round 15 — Operations, release, and buyer-facing readiness

The release gate is broad, and the packet workflow blocks external claims by default for sample/internal data. The next maturity leap is not more synthetic testing; it is a real external corpus, a real counterparty/reviewer workflow, and published anchors.

Assessment: research-grade posture is credible under the repository’s own definitions. Production-ready/A+ should remain blocked until live external evidence is present.

Recommended action: create a release checklist section for “external reliance packet” that requires real corpus manifest, provider export manifest, T1 manifest, published chain/registry anchors, outside review, buyer feedback, legal review, and regenerated FinOps report.

## Concrete code-change backlog

### 1. Strict receipt identity validation

Add helper logic to both private and public verification paths:

- `seq` must be exactly 1-based physical order in public attestations;
- `hash_version >= 6` links must carry a non-empty `receipt_id`;
- `receipt_id` values must be unique;
- v6 mint-log rows should have `receipt_id == f"KRY-{line_number:08d}"` unless a documented legacy migration path says otherwise;
- a promotion with `supersedes` should reference an existing non-zero source receipt, or be reported as a verification error rather than a silent no-op.

Suggested tests:

- `test_private_verify_chain_rejects_duplicate_v6_receipt_id`
- `test_private_verify_chain_rejects_wrong_v6_receipt_id_sequence`
- `test_public_verifier_rejects_non_monotonic_seq`
- `test_public_verifier_rejects_duplicate_receipt_id`
- `test_public_verifier_rejects_promotion_with_missing_supersedes_target`
- package-vs-standalone drift test for every new error case

### 2. Mint failure surfacing

`mint()` currently returns `None` for both invalid inputs and internal failures. Split these paths:

- invalid/non-positive tokens can continue returning `None`;
- corrupt decay state, lock failure, JSON failure, append failure, or fsync failure should log an exception and return a structured failure or raise a domain-specific exception;
- failure to update the mutable ledger after appending a valid mint should return a warning object or log loudly, because the chain and ledger diverge until reconciliation.

### 3. Claims wording hardening

Recommended wording changes:

- Replace “proves WHICH efficiency event occurred” with “proves the chain contains a claimed efficiency event and binds its declared magnitude/tier.”
- Replace “trustless two-party transfer protocol” with “federated verifier-mediated settlement protocol.”
- Replace “external token” with “external settlement/proof surface” unless immediately followed by the non-transferable/non-tradeable scope.

### 4. External price-source manifest

Create `docs/price_basis/PRICE_BASIS_2026_06_03.json` or equivalent, then bind its hash into artifacts. Include model key, USD/M, quality (`list`/`estimate`), source label, retrieval date, and notes. Keep the Python table generated from this manifest or add a drift test.

### 5. External reliance packet gate

Add a gate that refuses `external_verified_savings=allowed` unless the artifact includes:

- provider export manifest;
- corpus manifest;
- T1 manifest;
- published chain anchor reference;
- published registry anchor reference if settlement is claimed;
- outside review;
- buyer feedback;
- legal review;
- regenerated FinOps report bound to the same artifact hash.
