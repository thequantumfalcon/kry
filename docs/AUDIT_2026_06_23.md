# kry — Independent Repository Audit

**Date:** 2026-06-23
**Scope:** Full repository (`thequantumfalcon/kry`, branch `claude/repo-audit-ipkt88`), 245 tracked files, ~34.5k LOC Python.
**Method:** Static review of the security-critical trust path (mint chain, external verifier, attestation, TEE/SNP/TLSNotary verifiers, PQC signer, settlement/locks), plus build/CI/supply-chain and docs-vs-code claim verification. Test suite, linter, end-to-end demo, and committed evidence bundles were executed.

---

## 1. Executive summary

This is a **high-quality, unusually self-honest codebase**. The central design tension — "a hash chain proves *integrity*, not *veracity*" — is confronted head-on rather than papered over, and the code is consistently *more conservative than its own marketing*. The external verifier (`scripts/kry_verify.py`) genuinely imports nothing from the package, independently re-derives every number, reproduces the promotion overlay, and prints loud warnings about exactly what it cannot prove. Prior audit rounds (R5/R6/R14/R22, S1–S9, F1–F5, HOLE #7–#27) are documented inline and have real regression tests behind them.

**No directly exploitable cryptographic bypass was found.** Every field the hardware-attestation verifiers act on lies inside a signed region, the signature is verified with a hard-pinned algorithm against an operator-pinned root, and the accounting layer fails closed on tamper/NaN/negative inputs. The remaining findings are **defense-in-depth, fail-open-on-misconfiguration, and documentation-drift** issues — appropriate for a mature project, none rated higher than MEDIUM.

## 2. What is done exceptionally well

- **Honest-by-construction verifier.** `kry_verify.py` is stdlib-only, displays *recomputed* figures (never the operator's declared fields), and emits `anchor check: NONE — ⚠ OPERATOR-ASSERTED` whenever an anchored fraction is claimed without a published anchor. A `test_external_verify` drift-guard pins its constant tables to the package source so they cannot silently diverge.
- **Fail-closed defaults across the accounting layer.** `_finite_number` + `allow_nan=False` + `parse_constant` rejection close the NaN/Inf/negative gaming vectors everywhere; `value_multiplier` floors unknown models at 0.05 (not full credit); free-tier "savings" earn 0.
- **Atomic, durable persistence.** Every writer uses `tempfile.mkstemp` in the same dir + `flush` + `fsync` + `os.replace`, with monotonic `{count, tip}` checkpoints to catch truncation/rollback that a chain replay alone misses.
- **Hardware-attestation verifiers fail closed on missing crypto**, verify the signature *before* trusting any field, bind issuer identity via `verify_directly_issued_by`, pin the root by DER byte-equality, and confirm `report_data == SHA-512(measurement_json)` (SNP). All surfaced SNP fields were checked to lie within the signed `report[0:0x2A0]` region.
- **Threshold PQC soundness.** `t−1` valid signatures cannot reach quorum; the stranger verifier independently recomputes key fingerprints and rejects duplicate keys (single-key-relabel defense) and degenerate thresholds (`True`/`0`/float).
- **Supply chain is genuinely hardened.** Every GitHub Action is pinned to a 40-char SHA, workflows are least-privilege with no `pull_request_target`, no untrusted `${{ }}` shell interpolation, harden-runner egress auditing, and a hard release gate + SLSA build-provenance attestation. The custom PEP 517 backend executes no code at install time (the editable `.pth` is a bare path, not an `import` line).
- **Adversarial test discipline.** Malicious-packet fixtures, `test_independent_audit.py`, and `test_public_claims.py` actively guard against both exploit regressions and *marketing-claim drift*.

## 3. Findings

Severity reflects realistic impact in the documented threat model. None are release-blocking; all are worth addressing.

### MEDIUM

**M1 — PQC verifier: `alg` is attacker-controlled, un-allowlisted, and crashes instead of failing closed.**
`kry_pqc/verify.py:82,117`. The algorithm is read from the attacker-supplied `.sig.json` (`alg = artifact["alg"]`) and passed straight to `oqs.Signature(alg)` *outside* the parse guard (lines 78–92). **Confirmed empirically:** a bogus `alg` yields an uncaught `MechanismNotSupportedError` traceback rather than the documented `RESULT: FAILED` / exit 1 — and the `[PASS] message digest` line prints first, which is misleading. With a pinned `--public-key` this is not a forgery (mechanism key-lengths differ, so verification fails), but `alg` is never bound to the key fingerprint and there is no allowlist.
*Fix:* validate `alg` against `{"ML-DSA-44","ML-DSA-65","ML-DSA-87"}` inside the guard; bind `alg` into the fingerprint (`sha256(alg+b"\x00"+pk)`) so a pinned fingerprint also pins the mechanism.

**M2 — Nitro COSE `alg` guard fails open on an absent/malformed protected header.**
`scripts/kry_tee_verify.py:330-335`. `prot_hdr.get(1) not in (COSE_ALG_ES384, None)` treats a missing alg (`None`) and an undecodable protected header (`{}`) as acceptable. Backstopped by the hard-coded `ec.ECDSA(hashes.SHA384())` verify, so it is **not a live bypass**, but the documented guard does not fail closed.
*Fix:* require `prot_hdr.get(1) == COSE_ALG_ES384` exactly and treat a non-dict/undecodable protected header as a hard failure.

**M3 — `settle()` skips the double-spend ceiling for any grant with the `attested_balance == -1` sentinel.**
`src/kry/kry_settlement.py:108,814`. The authoritative commit-time double-spend recheck is gated on `if grant.attested_balance >= 0:`. A grant constructed outside the blessed `verify_and_accept` path (which sets a real ceiling) carries `-1.0` and bypasses the check entirely, settling an arbitrary `accepted_kry`. This is a *trust-the-caller* gap (settle is operator-side), not a remote exploit, but nothing authenticates a grant's provenance.
*Fix:* reject `attested_balance < 0` at `settle()`, or sign the grant in `verify_and_accept` and verify it in `settle()`.

**M4 — Settlement `self_asserted` conservation basis trusts the debit callback.**
`src/kry/kry_settlement.py:882-888`. When `a_balance_after_fn` is `None`, conservation is `True` by definition — A's real ledger movement is never independently observed, so a lying/under-reporting `debit_a_fn` can record a settled obligation with no offsetting credit. The receipt honestly *labels* this `self_asserted`, but the registry still records it.
*Fix:* require the `measured` basis (`a_balance_after_fn`) for any settlement that writes to the registry.

**M5 — `kry_pending._load` resets to `{}` on a present-but-corrupt store (fail-open).**
`src/kry/kry_pending.py:84-94`. A tampered/corrupt pending file is swallowed to `{}`, which erases the write-ahead `confirm` idempotency state for co-resident pendings — a narrower re-mint window (still capped by mint-side decay). Contrast `kry_mint._decayed_tokens:351-357`, which correctly *fails closed* on a present-but-unparseable file.
*Fix:* distinguish "file absent" (→ `{}`) from "file present but unparseable" (→ raise).

**M6 — Cross-process lock is a silent no-op when neither `fcntl` nor `msvcrt` is available.**
`src/kry/_locks.py:68`. On such a platform the cross-process serialization for settlement, reputation, and decay RMW vanishes with no log line. Latent on Linux/macOS (fcntl present), but it is a fail-*open* default.
*Fix:* emit a one-time `logger.warning` when the no-op path is taken.

**M7 (docs) — `KRY_TOKEN_SPEC.md` §3 contradicts the code on `cache_creation`.**
Spec lists `Cache creation | 0.1`; code uses **0.0** (`src/kry/kry_token.py:189`, `kry_mint.py:71`) to avoid double-counting a cache write against its later hit. The spec — cited as the authoritative unit definition — overstates credit. (The code is the more-honest side.)
*Fix:* set the spec row to `0.0`/remove it, add the missing `continuity_capsule: 0.1`, and refresh the spend table (it omits `opus-4.7-fast`, `deepseek-v4-flash`, `deepseek-r1`, `fireworks/*`).

### LOW

- **L1 — Cross-node double-spend lease ships default-OFF.** `kry_settlement.py:526-532`. With `KRY_SETTLE_LEASE_DIR` unset, two nodes can settle the same attested balance against unmerged registries and both succeed. Extensively documented ("HOLE D corollary"), but the default multi-node config is double-spendable. *Consider failing closed (refuse cross-node settlement) when no lease dir is configured.*
- **L2 — PQC secret key chmod-after-write TOCTOU.** `kry_pqc/signer.py:93-95` writes the key then narrows to `0o600`; a brief umask-default-readable window exists. *Fix:* `os.open(path, O_WRONLY|O_CREAT|O_EXCL, 0o600)` before writing.
- **L3 — PQC signatures lack domain separation / context binding.** `kry_pqc/{signer,threshold}.py` sign raw attestation bytes, so a single-signer signature is a valid threshold "contribution" over the same bytes and a contribution is replayable across councils that share a member. *Fix:* sign `SCHEME_TAG || policy_digest || bytes`.
- **L4 — CBOR decoder: duplicate map keys are last-wins; semantic tag number discarded.** `scripts/kry_tee_verify.py:139-147,188-190`. Not exploitable (the whole payload is signature-bound), robustness only. *Fix:* reject duplicate keys.
- **L5 — Release frontend unpinned.** `.github/workflows/release.yml:41` `pip install --upgrade pip build` on the runner holding `id-token: write`. *Fix:* hash-pin the build toolchain.
- **L6 — Wheel not byte-reproducible.** `build_backend.py:120-123` uses current mtimes. *Fix:* honor `SOURCE_DATE_EPOCH`.
- **L7 — PoC enclave non-reproducible.** `poc/nitro/enclave/Dockerfile:4,10` use floating base tags and there is no committed `Cargo.lock`, so the enclave PCR0 measurement is not reproducible. *Fix:* pin base images by `@sha256:` digest and commit the lockfile.
- **L8 (docs) — `KRY_READINESS.md:31` off-by-one** ("13" vs the code's `implemented: 14`).
- **L9 (docs) — Veracity ladder understates the floor inputs.** The README/`KRY_VERACITY_BINDING.md` ladder presents `veracity_floor` as "external anchor (T1+T2) only," but `_ANCHORED_TIERS` (`kry_mint.py:104`) also counts `holdout_validated`, an operator-run population estimate. The code's inline comments and the operator-view label are careful about this; the headline table should match.

### NITS

- `.gitignore:8-10` has a dead `!.env.example` negation (no such file) and omits `dist/`.
- `poc/nitro/parent_client.py:18` `int(sys.argv[1])` throws a raw traceback on bad input.
- `kry_pqc/verify.py:108` fingerprint pin accepts any prefix length (a 1–2 char `--expect-fingerprint` is brute-forceable); add a minimum-length warning.
- SNP verifier has no default ARK fingerprint pin (unlike the Nitro root default); operator-supplied ARK is anchor-on-trust (documented).

## 4. Prioritized recommendations

1. **M1** — allowlist + fingerprint-bind the PQC `alg`, and move its construction inside the parse guard (one-line robustness + clean fail-closed). *(confirmed crash today)*
2. **M3 / M4** — reject the `attested_balance == -1` sentinel at `settle()` and require the `measured` conservation basis for registry-recording settlements — the two concrete single-host accounting gaps.
3. **M2 / M6 / M5** — make the Nitro alg check and the lock no-op and the pending-load corruption case all *fail closed/loud* rather than open/silent.
4. **M7 / L8 / L9** — reconcile the three doc-vs-code drifts. For a project whose entire thesis is verifiable honesty, a stale spec table is a disproportionate reputational risk.
5. **L1** — decide whether default-off cross-node lease should instead fail closed.
6. **L5–L7** — tighten release/PoC reproducibility (pin build frontend, `SOURCE_DATE_EPOCH`, digest-pin enclave bases).

None of these block a `0.1.0` research-grade release. The codebase's honesty discipline is its strongest asset; the highest-value fixes are the ones that keep the *documentation* as rigorous as the *code*.
